## 5. Readiness, inspection, and observation contracts

`CapabilityReadiness` is exactly `ready`, `ready_empty`, `missing`, `malformed`, `version_mismatch`, and `unsupported`. `CapabilityInspection` stores only identity, expected/observed safe versions, readiness, safe source kind, strict `content_empty`, and safe issue codes. It stores no chart, content, producer, exception, traceback, path, object representation, or telemetry.

`CapabilityFactState` is exactly `present`, `absent_entity`, `capability_unavailable`, `malformed_capability`, `version_mismatch`, and `unsupported_capability`. `CapabilityFactObservation` stores canonical capability/entity identity, a strict `value_present` flag, canonical value only for `present`, and safe issues.

The implementation never uses truthiness to decide fact presence. Explicit `False`, integer/float zero, empty string where a capability vocabulary permits it, and empty canonical collections can be present values. Valid house 1â€“12 is present. A valid planet collection with no requested canonical planet yields `absent_entity`; unavailable or malformed collections yield capability states instead.

## 6. Read-only adapters and representation preservation

`inspect_capability` and `observe_capability_fact` accept AstroState-like objects and typed requirements. They perform attribute/mapping/shape reads only. They do not normalize, enrich, cache, attach a manifest, access filesystem/network/environment/time/randomness, or invoke any producer.

Adapter source kinds include `normalized_planets`, `planet_house_fields`, `legacy_basic_conjunction_list`, `legacy_whole_sign_graph`, `astro_lagna_sign`, `derived_functional_roles`, `legacy_planet_flags`, `legacy_metadata_exaltations`, and `legacy_exaltation_sources`.

Aspect preservation evidence:

- a list under `enrichments['aspects']` can satisfy only `aspects.basic_conjunction_list`;
- a mapping envelope with an `edges` list/tuple can satisfy only `aspects.whole_sign_graph`;
- explicit empty list and empty edges are independently `ready_empty`;
- a list presented for a graph requirement, or a graph for a list requirement, is `missing` with `representation_mismatch`;
- `config_version` remains safe producer metadata and is not compared by string coincidence to the capability contract;
- a valid legacy graph without a formal capability version uses adapter contract 1.0.0 and source kind `legacy_whole_sign_graph`;
- no conversion, overwrite, conjunction inference, target-none decision, or Yoga change occurs.

Functional-role policy: only `astro.derived['functional_roles']` is inspected as explicit prepared facts. Missing prepared facts remain missing. Empty valid prepared facts are ready-empty. `compute_functional_roles` and its tables are never called.

Exaltation policy: legacy planet `flags['exalted']` and `metadata['exaltations']` are independently recognized and retain distinct source kinds. False, zero, degree, and true values remain present. Malformed sources are rejected. Conflicting overlapping sources produce malformed `conflicting_sources`; no preference or astrology semantic interpretation is selected.
