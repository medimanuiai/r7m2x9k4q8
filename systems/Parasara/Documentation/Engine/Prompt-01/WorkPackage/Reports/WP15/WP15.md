## 5. Models, fields, invariants, and serialization

`CareerFactKind` has exactly five executed groups: `strong_in_house`, `house_lord_status`, `rajayoga_compatibility`, `base_kendra_strength`, and `tenth_house_occupant_strength`.

`CareerFactResult` stores fact ID/version/kind, strict `matched`, WP02 status, frozen inputs/evidence, ordered WP02 errors/traces, optional canonical `PredicateResult`/`ConditionResult`, and optional duration. Matched is true exactly for `MATCHED`; factual statuses have no errors; every non-factual status requires a typed safe error.

`CareerCandidateDefinition` stores exact candidate/type/version/source identity, frozen parameters, recursively ordered immutable compatibility context, base/matched/unmatched score metadata, and source index. It retains no raw YAML mapping, loader, registry, provider, callable, or public output.

`CareerCandidateEvaluation` stores its definition, complete fact, matching status, legacy adjusted score, effective contribution, frozen compatibility evidence, stable trace lineage, and optional telemetry. Unmatched/non-factual contribution must be zero.

`CareerEvaluationBatch` stores schema/evaluator version, prepared digest, every source-ordered candidate, typed base/component facts, base score, fixed denominator, frozen completeness, batch errors, and optional telemetry. Source indices must be contiguous, candidate IDs unique, and denominator equal candidate count.

WP03 canonical JSON is strict compact UTF-8, sorted for identity, rejects duplicate keys/non-finite values/unsupported objects, and uses lowercase SHA-256. Logical identity excludes all telemetry. Career capture normalizes optional backing predicate cache/duration telemetry to false/none, so logical and full Career persistence are both cold/warm, fresh-process, and cross-version identical. Public mapping order is separately retained in a recursively ordered immutable compatibility value and reconstructed only at projection.
