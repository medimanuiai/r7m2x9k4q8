## 3. Exact `PredicateDefinition` contract

The locked field names were used without reconciliation or speculative additions.

| Field | Stored type | Validation |
|---|---|---|
| `predicate_id` | `str` | uppercase ASCII `^[A-Z][A-Z0-9_]*$`; no whitespace/lowercase/operator |
| `predicate_version` | `str` | Stage-01 SemVer `MAJOR.MINOR.PATCH`; decimal components have no illegal leading zero |
| `description` | `str` | required, non-empty, trimmed |
| `parameter_schema` | WP03 `FrozenMapping` | required immutable descriptive metadata; no runtime enforcement |
| `required_capabilities` | `tuple[str, ...]` | immutable, unique stable lowercase identifiers in lexicographic order |
| `cacheable` | actual `bool` | no integer substitute; cannot be true when deterministic is false |
| `deterministic` | actual `bool` | no integer substitute |
| `cost_class` | string-valued `CostClass` enum | exactly `low`, `medium`, or `high` |
| `system_scope` | `str` | stable lowercase identifier; production uses `parasara` |
| `deprecated` | actual `bool` | defaults false |
| `replacement` | `str | None` | canonical ID only; not self/alias/operator; target and cycles checked at finalization |
| `aliases` | `tuple[str, ...]` | unique canonical-format IDs; no self/operator/collision; deterministic order |
| `handler` | callable | required and stored without a wrapper |

The dataclass is frozen, slotted, keyword-only, and has exactly these thirteen fields. Nested schemas use the single WP03 `FrozenMapping`; capabilities and aliases use tuples. Unknown metadata fields fail construction.

## 7. Registry lifecycle and isolation

`PredicateRegistry` has a build phase and a finalized phase:

- build accepts only validated definitions;
- validation/collision/handler checks complete before state mutation;
- duplicate canonical IDs, canonical/alias collisions, alias/alias collisions, and duplicate canonical registration of one handler fail;
- aliases are vocabulary on one definition, not duplicate definitions;
- finalization validates replacement targets and cycles;
- `register`, `remove`, `replace`, and `reset` fail explicitly after finalization;
- readiness and frozen state are queryable;
- enumeration is sorted independently of registration order;
- canonical definitions are returned as tuples and metadata snapshots as `FrozenMapping`;
- no public mutable dictionary or reset hook exists.

Tests cover invalid-registration atomicity, build operations, every post-freeze mutation, missing/alias replacement targets, cycles, scoped failure restoration, and isolation from production.

## 8. Deterministic bootstrap and compatibility view

`bootstrap_production_registry()` uses four explicit internal states: uninitialized, building, ready, and failed. It constructs locally and publishes only after all five definitions finalize. Recursive/partial initialization and prior failure raise explicit errors; failed bootstrap publishes `None`. Repeated successful calls return the same ready frozen instance.

Bootstrap does not enumerate files, discover plugins, scan modules, depend on Yoga, or depend on a prior predicate import. Fresh-process tests cover registry-first, predicate-first, and Yoga-first import orders. Another fresh-process test imports only the legacy engine, proves the handler module is not yet loaded, and evaluates all six exposed IDs successfully.
