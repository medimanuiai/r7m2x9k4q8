## 3. Exact prepared contracts

### PreparedPlanet

Exact fields are `planet_id`, `house`, and `sign`.

- ID is one of the nine WP05 canonical planets.
- House is an exact non-Boolean integer 1–12 or `None`.
- Sign is a canonical repository sign or `None`.
- Planets are stored in Sun, Moon, Mars, Mercury, Jupiter, Venus, Saturn, Rahu, Ketu catalog order with absent planets omitted.
- Duplicate canonical identities are fatal.
- Degree, flags, dignity, role, score, strength, vargas, provider values, and source indices are absent.

An invalid source house is represented as `house=None` on the safe planet projection while `planets.house_placement` is `malformed`; it is not converted to factual false and does not make the whole preparation fatal.

### PreparedCapability

Fields are `capability_id`, catalog `capability_version`, safe `observed_version`, `readiness`, `source_kind`, `content_empty`, canonical immutable `content`, and ordered safe `issues`.

- ready/ready-empty content is recursively frozen with WP03;
- ready requires nonempty content and matching observed/catalog versions;
- ready-empty requires explicit empty content and a WP06 ready-empty catalog policy;
- missing/unsupported claims no observed version, source, or empty content;
- unavailable states retain no factual content;
- version mismatch retains the safe observed SemVer and no content;
- all observed nonmissing entries retain a safe source kind;
- duplicate issue codes and malformed direct content are rejected;
- an unknown standalone model is valid only when explicitly `unsupported`;
- a prepared state itself contains exactly the seven finalized production catalog identities.

`PreparedCapabilityMapping` is a model-owned immutable, lexicographically ordered mapping because WP03 correctly refuses to reflect/freeze arbitrary dataclass instances. It stores only `PreparedCapability` values behind a read-only mapping proxy and exposes no mutable backing value.

### PreparedAstroState

Fields are `schema_version`, `producer_version`, `normalization_version`, `system_scope`, `lagna_sign`, canonical `planets`, and the complete immutable capability manifest.

Construction enforces:

- strict SemVer versions and `parasara` scope;
- canonical planet order and uniqueness;
- exactly the finalized seven-capability manifest;
- Lagna equality with the `chart.lagna` capability;
- prepared planet equality with `planets.normalized` content;
- ready placement equality with all prepared planet houses.

No source fingerprint was added because no safe provider-independent authoritative identifier exists.

### Preparation outcome

`PreparationOutcome` contains exact `succeeded`, `state`, and immutable typed `issues`. `PreparationIssue` contains safe `code`, rooted safe `path`, and optional canonical `capability_id`; it contains no raw value, exception, traceback, object representation, or provider detail.

Ordinary missing, ready-empty, malformed, version-mismatch, and unsupported readiness is inside a successful state. No-state outcomes are limited to invalid input/contracts, duplicate canonical planet/supply identity, unknown supplies, unsafe/cyclic/unsupported canonical content, contradictory model invariants, or unexpected construction defects.

## 6. Explicit-supply policy

`prepare_predicate_state(astro, *, capability_supplies=None, versions=None)` accepts immutable typed `CapabilitySupply` objects in a mapping or sequence. Each declares capability ID, supplied version, source kind, and already-computed content.

- inputs are traversed read-only and content is defensively frozen/projected;
- supply ordering is canonical and independent of input mapping insertion order;
- duplicate supplies, mismatched mapping keys, invalid containers, and unknown catalog identities fail safely;
- a supplied version mismatch becomes a version-mismatch manifest entry;
- malformed known content becomes a malformed manifest entry;
- valid empty content follows the WP06 capability empty policy;
- a supply replaces only a missing same-capability compatibility location;
- any present/ready-empty/malformed/versioned compatibility location plus a same-capability supply becomes deterministic malformed `conflicting_supply` with no selected content;
- no callback, lazy value, path, filesystem source, or producer is accepted.

## 8. Typed evaluation context

`PredicateEvaluationContext` is frozen and contains only:

- `system_scope = parasara`;
- `evaluation_mode = EvaluationMode.DEFAULT` (`default`);
- `selected_planets`: `None` for all eligible, empty tuple for explicitly none, or a normalized canonical-order tuple;
- `evaluation_instant`: optional explicit timezone-aware UTC `datetime`.

Selected planet duplicates are rejected after WP05-style case/trim normalization. A supplied instant must have UTC offset zero and serializes exactly as `YYYY-MM-DDTHH:MM:SS.ffffffZ`. There is no system-time fallback. Current registry requirements contain no instant dependency and no current handler/evaluator was redirected to this context.
