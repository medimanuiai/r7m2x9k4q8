## 4. Executable architecture policy

`tests/wp17/test_architecture_policy.py` parses executable Python ASTs. Reports,
prompts, archives, fixtures, string-only negative tests, generated environments,
and unrelated systems are outside its scope.

Enforced rules:

- retired runtime/engine/predicate/instrumentation modules stay absent;
- retired raw helpers, `RuleMatch`, runtime evaluators, and instrumentation calls
  stay absent;
- production direct calls to the five registered handlers stay absent outside
  the sole `PredicateEvaluator` invocation boundary;
- production never imports `tests` or test helpers;
- predicate preparation/evaluation cannot import interpreters, Yoga public
  paths, tools/snapshots, raw Surya, or tests;
- Yoga typed evaluation cannot import Career/interpreter/public/API layers;
- Career factual/model code cannot import Yoga output, generic loaders, raw
  Surya, tools/snapshots, or tests;
- core canonical/model serialization cannot import clocks, randomness,
  environment/CWD, filesystem, subprocess, or networking modules;
- factual typed functions cannot advertise raw Boolean, tuple, or dictionary
  returns;
- Yoga/Career compatibility projectors cannot be called from typed factual
  evaluators.

Narrow exceptions are explicit:

- `perf_counter_ns` is allowed only in predicate/condition telemetry; logical
  serialization tests prove it cannot change identity or public output;
- `yoga_engine.py` may use `Path`/YAML in its explicit source-loading function
  and deterministic UUID5 for compatibility trace references; the guarded
  `evaluate_yoga_batch` boundary performs neither I/O nor random identity;
- Yoga Aspect/role producers are allowed only in explicit preparation and are
  guarded against invocation from typed batch evaluation;
- Career's `PureWindowsPath` formats a locked repository-relative compatibility
  string and performs no filesystem access;
- retained generic/Yoga loaders exist only for the supported compatibility
  loader contract and are excluded from predicate, Yoga typed, Career factual,
  and tooling evaluation paths;
- direct handler calls remain allowed only in the pre-existing explicit handler
  unit tests. WP16 plus the new production-wide gate enforces zero production
  bypass.

## 10. Discovered violation and minimal production fix

The initial architecture test failed because
`systems/Parasara/engine/rules/canonical.py` imported `pathlib`, `socket`, and
`uuid` solely to recognize and reject path/socket/UUID objects. These imports
violated the locked core-serialization dependency boundary even though no I/O or
random generation occurred.

The minimal fix removed those three imports and added `_has_runtime_type`, which
recognizes the same rejected runtime types through class MRO module/name
metadata. It does not use `repr`, object ID, memory address, I/O, network, clock,
or randomness. Existing rejection categories and exception messages remain
unchanged.

Proof of behavior preservation:

- the initial focused architecture result was 5 passed, 1 failed at exactly
  those three imports;
- after the fix, the architecture gate passed;
- 173 existing predicate/condition/Yoga/Career serialization and typed-model
  tests passed immediately;
- all WP17 focused, 731 contract, and repeated full suites pass in both lanes;
- every Career, Yoga, tooling, coverage, approved-snapshot, and manifest hash
  remains exact.

No other production violation was found. No astrology, rule, score, schema,
snapshot, dependency, CI, or public-output change was made.
