## 3. Exact executable contracts

### `ParameterKind`

String enum with exactly:

- `house`;
- `planet`;
- `role_collection`.

### `ParameterSpec`

| Field | Type | Contract |
|---|---|---|
| `name` | exact lowercase identifier | canonical parameter key |
| `kind` | `ParameterKind` | executable normalization rule |
| `required` | actual Boolean | required versus optional |
| `aliases` | tuple of lowercase identifiers | explicit only; unique and lexicographic |
| `allowed_values` | tuple of strings | nonempty only for role collections |
| `has_default` | actual Boolean | distinguishes no default from an explicit default |
| `default` | canonical value | validated through the declared kind; forbidden on required specs |

### `ParameterSchema`

| Field | Type | Contract |
|---|---|---|
| `predicate_id` | canonical ID | must match its `PredicateDefinition` |
| `schema_version` | Stage-01 SemVer | must equal predicate version |
| `specifications` | ordered tuple of `ParameterSpec` | issue order and accepted vocabulary |

Schema construction rejects duplicate canonical names, alias/canonical collisions, alias/alias collisions, invalid alias ordering, unsupported kind/constraint combinations, invalid defaults, logical-operator IDs, wrong versions, and mutable spec containers.

### Validation issue and outcome

`ParameterValidationIssue` fields are exactly:

```text
predicate_id, parameter_name, code, expected, path
```

Supported stable codes are:

```text
invalid_container, missing_required, unknown_parameter, invalid_type,
invalid_value, conflicting_alias, duplicate_value
```

`ParameterValidationOutcome` fields are exactly:

```text
valid, normalized_inputs, issues
```

A success has a `FrozenMapping` and no issues. A failure has no normalized inputs and a nonempty immutable issue tuple. Contradictory states fail construction.

## 5. Planet policy

The exact Stage-01 catalog is:

```text
Sun, Moon, Mars, Mercury, Jupiter, Venus, Saturn, Rahu, Ketu
```

An input must be an actual string. Outer whitespace is trimmed; ASCII case is normalized to the catalog spelling. Empty, non-ASCII transliteration, abbreviation, unknown planet, `Ascendant`, `Lagna`, outer planet, number, Boolean, `None`, and custom object fail. No fuzzy matching or aliases exist. Validation does not inspect AstroState, so a valid catalog planet remains parameter-valid even if absent from a chart.

## 6. House policy

A house must have exact Python type `int` and be within 1–12 inclusive. Because exact type is required, Boolean and `IntEnum` are rejected along with floats, integral floats, numeric/nonnumeric strings, `Decimal`, custom numeric objects, `None`, zero, negatives, and values above 12. No coercion, clamping, or fallback occurs.

## 7. Aspect, key, alias, and default policies

An empty Aspect mapping is valid and means no filter, preserving “any available aspect edge” compatibility for later migration. Each filter is independently optional. Omitted filters remain absent from normalized inputs; explicit `None` is invalid. All four filters together are valid after strict house/planet normalization.

The top-level input must implement `Mapping`; `None`, falsey non-mappings, list/tuple/string/number/generator, dataclass, Pydantic model, and custom object fail with one safe container issue. Mapping read failure is converted to the same safe issue without raw exception text.

Keys must be actual exact lowercase schema strings. They are not trimmed or case-normalized. Unknown, misspelled, whitespace-padded, wrong-case, and non-string keys fail. Unknown string keys are processed in lexicographic order but are represented with a redacted safe unknown path.

Production declares zero parameter aliases. Synthetic schemas prove exact alias success, canonical-plus-alias conflict, alias/alias and alias/canonical collision rejection, alias-before-unknown processing, and deterministic conflict handling. Synthetic optional defaults prove central validation and canonical freezing; production defines no defaults.

## 9. Safe failure and error-detail contract

Issue order is:

1. schema specifications in declared order;
2. alias conflicts at their canonical specification position;
3. unknown string keys processed lexicographically;
4. redacted non-string unknown keys;
5. one container issue first and alone when the top-level input is not/readably not a mapping.

Issues contain only canonical predicate ID, canonical parameter name when known, stable reason code, safe expected-kind/catalog identifier, and logical path. Unknown names are redacted as `$.parameters[unknown]`. No raw supplied value, mapping, exception, traceback, class representation, source content, secret, or memory address is retained.

The error adapter produces exactly:

```text
code = invalid_parameters
message = Predicate parameters are invalid.
predicate_id = canonical predicate ID
details = FrozenMapping containing ordered immutable issue details
recoverable = True
```

It rejects successful outcomes and predicate-ID disagreement. It constructs no `PredicateResult` and invokes no handler.
