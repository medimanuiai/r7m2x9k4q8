## 4. Canonical API

Location: `systems/Parasara/engine/rules/planet_in_house.py`.

```python
def evaluate_planet_in_house(
    parameters: Any,
    state: PreparedAstroState,
    context: PredicateEvaluationContext,
) -> PredicateResult
```

The seam is fixed to `PLANET_IN_HOUSE`; on ordinary execution it obtains the authoritative `PredicateDefinition`, version `1.0.0`, WP05 schema, and sorted required capabilities from `get_production_registry()`. It rejects a non-prepared state or untyped context at the canonical boundary and safely returns canonical `status=error`. It never accepts mutable `AstroState` as factual input.

`PredicateEvaluationContext.selected_planets` is intentionally factually neutral for this single-entity predicate. `None`, empty, and nonempty selections produce the same logical result. No new selection semantic was invented.

## 5. Parameter normalization and validation matrix

| Input | Outcome |
|---|---|
| canonical planet plus house 1–12 | accepted |
| planet outer whitespace/case variant | trimmed and normalized to one of the nine WP05 canonical IDs |

Invalid results use `inputs={}`, `evidence={}`, `matched=false`, and the existing WP05 `invalid_parameters_error`. No capability or fact query executes. Successful results contain only canonical normalized `planet` and strict integer `house` inputs.

## 6. Capability requirement and readiness matrix

The definition supplies these requirements in deterministic capability-ID order:

1. `planets.house_placement@1.0.0`;
2. `planets.normalized@1.0.0`.

Both are inspected through `inspect_prepared_capability`; facts are read only through `observe_prepared_planet` and `observe_prepared_planet_house`.

| Inspection | Predicate status | Error code | Recoverable |
|---|---|---|---:|
| `ready`, nonempty, version-compatible | continue | — | — |
| `missing` | `missing_capability` | `missing_capability` | true |
| `malformed` | `missing_capability` | `malformed_capability` | false |
| `version_mismatch` | `missing_capability` | `capability_version_mismatch` | true |
| `unsupported` | `missing_capability` | `unsupported_capability` | false |
| prohibited `ready_empty` | `missing_capability` | `ready_empty_not_allowed` | true |

Every state is tested independently for each required capability by controlled inspection substitution. Multiple failures retain registry order. Any capability failure stops before entity or house queries, so malformed placement content cannot be partially used and no mutable-data fallback exists.

## 7. Factual truth table

| Observation after valid parameters/capabilities | Status | Matched | Error |

| Step ID | Operation | Parent |

| `planet_in_house.execution` | `canonical_execution` | last completed step; unexpected defects only |

Trace order is semantic and left-to-right. The decisive failed validation, capability, planet, house, or execution step carries its typed `error_code`. Final disposition records only status and matched state.

The no-fabrication policy is early termination: invalid parameters emit only parameter validation and final disposition; capability failure emits parameters, capabilities, and final disposition; entity/fact failures stop at the relevant factual step. Skipped steps are absent rather than fabricated.
