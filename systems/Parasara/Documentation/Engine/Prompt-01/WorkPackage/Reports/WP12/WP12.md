## 5. Exact immutable model inventories and invariants

`DEFINITION_SCHEMA_VERSION` is `1.0.0`.

### `RuleSourceIdentity`

1. `source_name`
2. `rule_id`
3. `rule_index`

`source_name` is a safe logical relative identity, never an absolute/CWD-derived path. Traversal, drive/URI syntax, backslashes, blank segments, and surrounding whitespace are rejected. `rule_id` is optional stable syntax; `rule_index` is optional, non-Boolean, integer, and nonnegative.

### `DefinitionIssue`

1. `code`
2. `message`
3. `node_path`
4. `source`
5. `predicate_id`
6. `parameter_name`
7. `details`
8. `severity`

Severity is exactly `DefinitionIssueSeverity.ERROR` (`error`). Every code has exactly one fixed message. Details are recursively frozen, bounded to safe keys/scalars/tuples, and reject whitespace-bearing exception text, paths, mappings containing supplied fragments, unsupported values, and excessive size/depth. Raw supplied parameter values, parser text, exception type/text/traceback, absolute paths, object identity, `repr`, secrets, and memory addresses are never projected.

### `ValidatedNodeBinding`

1. `node_id`
2. `node_path`
3. `node_kind`
4. `requested_type`
5. `canonical_type`
6. `predicate_version`
7. `parameters`
8. `required_capabilities`
9. `child_count`

`node_id == node_path` and uses WP10 path derivation. Kind is exactly `logical` or `predicate`. Logical bindings contain a canonical operator and declared child count but no predicate version/parameters/capabilities. Predicate bindings contain canonical SemVer, normalized `FrozenMapping` parameters, ordered `(capability_id, version)` tuples, and no child count. Bindings are emitted in declared preorder.

### `ValidatedConditionDefinition`

1. `source`
2. `normalized_condition`
3. `node_bindings`

The normalized mapping is recursively immutable, root type agrees with the unique root-first binding, and bindings have unique paths.

### `DefinitionValidationOutcome`

1. `valid`
2. `definition`
3. `issues`

A valid outcome contains exactly one definition and no issues. An invalid outcome contains no definition and a nonempty ordered issue tuple.

### Strict Yoga models

`ValidatedYogaRule` contains `rule_id`, `source`, an immutable current Yoga `rule` with normalized conditions, and its `condition` definition. `YogaRuleSetValidationOutcome` contains `valid`, ordered valid `rules`, and ordered `issues`; `valid` is true exactly when issues are empty. Caller mutation cannot reach either model.

All models are frozen, slotted, keyword-only dataclasses. Nested mappings/lists are copied into WP03 `FrozenMapping`/tuple values.

## 7. Structural/operator/safety policy

Only `AND`, `OR`, and `NOT` normalize by the existing trim/uppercase policy. Child order is never sorted or deduplicated. AND/OR require one or more children; NOT requires exactly one. Raw children must be a list. WP12's normalized immutable definition stores the same sequence as a tuple; WP10 admits that tuple only when its parent is the WP03 `FrozenMapping`, so a bypassed raw tuple remains rejected.

Every logical node has exactly `type`/`children`; every leaf exactly `type`/`params`. Null/wrong containers, conflicts, alternate/misspelled fields, and unsupported operators are errors.

WP10's shared pure preflight enforces:

- root depth 1;
- maximum depth 64;
- maximum total nodes 4096 including root;
- active-ancestry mapping/list cycle rejection;
- count before depth before cycle, matching WP10.

Depth 64 and 4096 nodes validate; depth 65 and node 4097 produce one root fatal issue. A cycle likewise produces one root fatal issue. Other issue order is deterministic preorder: node structural issue before descendants, siblings in declaration order, and WP05 issues in schema order.
